# 클라우드 서비스로서의 Adobe Experience Manager 아키텍처 소개 {#an-introduction-to-the-architecture-adobe-experience-manager-as-a-cloud-service}

AEM(Adobe Experience Manager)을 클라우드 서비스로 설정하면 아키텍처가 변경되었습니다.

## 크기 조절 {#scaling}

이제 클라우드 서비스로 AEM에 다음이 포함됩니다.

* AEM 이미지 수가 다양한 동적 아키텍처입니다.

![동적](assets/concepts-01.png "아키텍처동적 아키텍처")

이 아키텍처:

* 실제 *트래픽과* 실제 ** 활동을 기반으로 크기를 조절합니다.

* 필요한 경우에만 실행되는 개별 인스턴스가 있습니다.

* 모듈식 애플리케이션을 사용합니다.

* 작성자 클러스터가 기본값으로 있음;이로 인해 유지 관리 작업의 중단 시간이 줄어듭니다.

이렇게 하면 다양한 사용 패턴을 자동으로 적용할 수 있습니다.

![다양한 사용 패턴을 위한](assets/concepts-02.png "자동 정렬다양한 사용 패턴을 위한 자동 처리")

이를 위해 클라우드 서비스로 AEM의 모든 인스턴스가 동일하게 생성됩니다. 각 인스턴스는 노드 수, 할당된 메모리 및 할당된 컴퓨팅 용량과 동일한 기본 크기 특성을 갖습니다.

클라우드 서비스로서의 AEM은 다음과 같은 오케스트레이션 엔진의 사용을 기반으로 합니다.

* 지속적으로 서비스 상태를 모니터링합니다.

* 실제 요구 사항에 따라 각 서비스 인스턴스를 동적으로 크기 조정두 가지 모두 적절하게 확대하거나 축소할 수 있습니다.

다음과 같습니다.

* 각 노드의 노드 수, 메모리 용량 및 할당된 CPU 용량에 적용됩니다.

* AEM 파섹

서비스의 임차인당 인스턴스 크기 조정은 두 축에서 자동 또는 수동일 수 있습니다.

* 세로:할당된 메모리 및 CPU 용량은 고정된 수의 노드에 대해 확대하거나 축소할 수 있습니다.

* 가로:지정된 서비스에 대한 노드 수를 늘리거나 줄일 수 있습니다.

## 환경 {#environments}

>[!NOTE]
>
> 자세한 내용은 배포 - [런타임 모드를 참조하십시오.](/help/implementing/deploying/overview.md#runmodes)

클라우드 서비스로 AEM을 사용할 수 있습니다. 각 인스턴스는 전체 AEM 환경을 나타냅니다. AEM을 클라우드 서비스로 사용할 수 있는 환경에는 다음 네 가지 유형이 있습니다.

* **프로덕션 환경**:비즈니스 전문가를 위한 애플리케이션을 호스팅합니다.

* **스테이지 환경**:는 항상 1:1 관계에서 단일 프로덕션 환경에 연결됩니다. 스테이지 환경은 애플리케이션 변경 사항이 프로덕션 환경에 푸시되기 전에 다양한 성능 및 품질 테스트에 사용됩니다.

* **개발 환경**:개발자는 스테이지 및 프로덕션 환경과 동일한 런타임 조건 하에 AEM 애플리케이션을 구현할 수 있습니다.

* **데모 환경**:평가, 데모, 프로토타이핑 및 트레이닝 용도로 사용할 수 있습니다.

개발 및 데모 환경을 *비프로덕션* 환경이라고 합니다.

## 프로그램 {#programs}

모든 새 AEM 프로젝트는 항상 하나의 특정 코드베이스에 연결되어 있으므로 프로젝트에 대한 구성 및 사용자 지정 코드를 모두 저장할 수 있습니다. 이 정보는 일반적인 Git 클라이언트를 통해 액세스할 수 있는 코드 리포지토리에 저장되며, 새 프로그램을 만들 때 사용할 수 있습니다.

AEM 프로그램은 다음을 포함하는 컨테이너입니다.

|  프로그램 요소 |  번호 |
|--- |--- |
| 코드 리포지토리(Git) |  1 |
| 기준 이미지(사이트 또는 자산) |  1 |
| 스테이지 및 프로덕션 환경 세트(1:1) | 0 또는 1 |
| 비프로덕션 환경(개발 또는 데모) | 0 - N |
| 각 환경을 위한 파이프라인 | 0 또는 1 |

AEM에서 처음 클라우드 서비스로 사용할 수 있는 프로그램은 두 가지가 있습니다.

* AEM Cloud 사이트 서비스

* AEM Cloud Assets Service

이 두 가지 모두 다양한 기능과 기능을 이용할 수 있습니다. 작성자 티어에는 모든 프로그램에 대한 모든 사이트 및 자산 기능이 포함되지만 자산 프로그램에는 기본적으로 게시 계층이 없습니다.

## 런타임 아키텍처 {#runtime-architecture}

이 새로운 아키텍처에는 다음과 같은 다양한 주요 구성 요소가 있습니다.

<!--- needs reworking -->

![클라우드 서비스로 AEM 사용 - 런타임](assets/concepts-03.png "아키텍처AEM을 클라우드 서비스로 사용 - 런타임 아키텍처")

* 클라우드 서비스로 AEM Sites의 경우:

   * 각 환경에 대한 제작 계층과 게시 계층의 개념이 계속 있습니다(높은 수준).

   * 작성자 계층은 단일 작성자 클러스터 내에 두 개 이상의 노드로 구성됩니다. 작성 활동에 따라 자동으로 크기가 조절됩니다.

      * 컨텐츠 작성자/작성자는 AEM 작성자 계층에 로그인하여 컨텐츠를 생성, 편집 및 관리합니다.

      * 작성자 계층에 로그인은 Adobe IMS(Identity Management Services)에서 관리합니다.

      * 자산 통합 및 처리는 전용 자산 계산 서비스를 사용합니다.
   * 게시 계층은 단일 게시 팜 내에 있는 두 개 이상의 노드로 구성됩니다.그들은 서로 독립적으로 작동할 수 있다. 각 노드는 AEM 게시자와 AEM Dispatcher 모듈이 장착된 웹 서버로 구성됩니다. 사이트 트래픽 요구 사항에 따라 자동으로 크기가 조정됩니다.

      * 최종 사용자 또는 사이트 방문자는 AEM 게시 서비스를 통해 웹 사이트를 방문합니다.


* 클라우드 서비스로 AEM 자산의 경우:

   * 아키텍처에는 작성 환경만 포함됩니다.

* 작성자 계층과 게시 계층 모두 Content Repository Service에서 컨텐츠를 읽고 유지합니다.

   * 게시 계층은 지속성 레이어의 컨텐츠만 읽습니다.

   * 작성자 계층은 지속성 레이어에서 컨텐츠를 읽고 씁니다.

   * Blob 스토리지는 게시 및 작성자 계층 전체에서 공유됩니다.파일은 *이동되지*&#x200B;않습니다.

   * 작성 계층에서 컨텐츠가 승인되면 활성화될 수 있으므로 게시 계층 지속성 레이어로 푸시됩니다. 미들웨어 파이프라인인 Replication Service를 통해 발생합니다. 이 파이프라인은 파이프라인으로 푸시된 컨텐트에 대한 개별 게시 서비스 노드를 구독하면 새 컨텐츠를 수신합니다.

      >[!NOTE]
      >
      >자세한 내용은 복제를 [참조하십시오](/help/operations/replication.md).

   * 개발자 및 관리자는 Cloud Manager를 통해 제공되는 CI/CD(Continuous Integration/Continuous Delivery) 서비스를 사용하여 AEM을 클라우드 서비스 [애플리케이션으로 관리합니다](/help/overview/what-is-new-and-different.md#cloud-manager). 여기에는 Cloud Manager의 CI/CD 파이프라인을 사용한 코드 및 구성 배포가 포함됩니다. 모니터링, 유지 관리 및 문제 해결과 관련된 모든 것(예: 로그 파일)은 Cloud Manager 내에서 고객에게 노출됩니다.

   * 작성자 및 게시 계층에 액세스하는 작업은 항상 로드 밸런서를 통해 수행됩니다. 이 기능은 항상 각 계층의 활성 노드에 최신 상태로 유지됩니다.

   * 게시 계층의 경우 CDN 파섹 서비스를 첫 번째 시작 지점으로 사용할 수도 있습니다.

* AEM을 클라우드 서비스로 데모하는 경우 아키텍처가 단일 작성자 노드로 간소화됩니다. 따라서 표준 개발, 단계 또는 제작 환경의 모든 특성을 제공하지는 않습니다. 또한 일부 다운타임이 있을 수 있고 백업/복원 작업을 지원하지 않는다는 것을 의미합니다.

## 배포 아키텍처 {#deployment-architecture}

Cloud Manager는 AEM 인스턴스에 대한 모든 업데이트를 클라우드 서비스로 관리합니다. 고객 애플리케이션을 구축, 테스트 및 배포할 수 있는 유일한 방법은 작성자와 게시 계층 모두에게 필수입니다. 이러한 업데이트는 새로운 버전의 AEM Cloud 서비스가 준비될 때, 또는 새로운 버전의 애플리케이션이 준비되면 고객이 트리거할 수 있습니다.

기술적으로, 이는 프로그램 내의 각 환경에 연결된 배포 파이프라인의 개념 때문에 구현됩니다. Cloud Manager 파이프라인이 실행되면 고객 애플리케이션의 새 버전이 생성되며, 작성자 및 게시 계층은 둘 다 사용됩니다. 이는 최신 고객 패키지와 최신 기준선 Adobe 이미지를 결합하여 달성됩니다. 새로운 이미지가 성공적으로 구축 및 테스트되면 Cloud Manager는 롤링 업데이트 패턴을 사용하여 모든 서비스 노드를 업데이트하여 최신 버전의 이미지로 자동 변환하는 작업을 완전히 자동화합니다. 이로 인해 작성자 또는 게시 서비스에 대한 다운타임이 발생하지 않습니다.

## 컨텐츠 배포 {#content-distribution}

Adobe Experience Manager as a Cloud Service는 콘텐츠 게시 방식을 수정했습니다. AEM을 클라우드 서비스로 사용하는 경우, 이전 버전의 AEM의 복제 프레임워크는 더 이상 페이지를 게시할 때 사용되지 않습니다(변경 내용을 작성자 인스턴스에서 게시 인스턴스로 이동).

이제 클라우드 서비스로 AEM에서 [콘텐츠 배포](https://sling.apache.org/documentation/bundles/content-distribution.html) 기능을 사용하여 적절한 콘텐츠를 이동합니다. 이 기능은 AEM 런타임 외부에 있는 Adobe I/O에서 실행되는 파이프라인 서비스를 사용합니다.

게시 노드가 런타임 중에 추가, 제거 또는 재생될 때 자동으로 자동 자체 구성을 포함하여 설정이 자동화됩니다.

단일 게시 또는 게시 취소 요청에는 여러 리소스가 포함될 수 있지만 모든 사람에게 적용된 단일 상태가 반환됩니다.AEM 게시 서비스의 모든 리소스에 대해 성공하거나 모든 리소스에 실패합니다. 이렇게 하면 AEM 게시 서비스 내의 리소스가 일관성 없는 상태가 되지 않습니다.

## 주요 발전 방안 {#key-evolutions}

클라우드 서비스로서 AEM을 위한 새로운 아키텍처는 이전 세대에 비해 몇 가지 근본적인 변화와 혁신을 제공합니다.

* 모든 파일(Blob)은 클라우드 데이터 저장소에서 직접 업로드되고 제공됩니다. 연관된 비트 스트림은 AEM 작성자 및 게시 서비스의 JVM을 거치지 않습니다. 따라서 AEM 작성자 및 게시 서비스의 노드는 크기가 더 작아질 수 있고 빠른 자동 처리 속도와 호환됩니다. 비즈니스 전문가는 이미지, 비디오 등을 업로드 및 다운로드할 때 더 빠른 경험을 제공합니다.

* 이제 컨텐츠 게시로 구성된 모든 작업에 구독 패턴을 따르는 파이프라인이 포함됩니다. 게시된 컨텐츠는 파이프라인의 다양한 대기열로 푸시되어 게시 서비스의 모든 노드가 구독됩니다. 따라서 작성자 계층은 게시 서비스의 노드 수를 인식할 필요가 없습니다.이렇게 하면 제작 계층의 자동 지정이 빨라집니다.

* 게시 노드의 라이프사이클을 자동화하기 위해 황금 마스터 개념이 도입되었습니다. 골든 마스터는 특수 게시 노드로서, 최종 사용자가 액세스할 수 없으며, 게시 서비스의 모든 노드가 생성되는 노드입니다. 압축과 같은 유지 관리 작업은 골든 마스터에 첨부된 컨텐츠 저장소에서 수행됩니다. 게시 노드는 매일 재활용되며 반복적인 유지 관리가 필요하지 않습니다.이전에는 이러한 유지 관리를 위해 특히 작성자 인스턴스에 대해 약간의 다운타임이 필요했습니다.

* 아키텍처는 애플리케이션 코드와 구성에서 애플리케이션 컨텐츠를 완전히 분리합니다. 모든 코드와 구성은 실질적으로 변경할 수 없으며 작성 및 게시 서비스의 다양한 노드를 만드는 데 사용되는 기준 이미지에 결합됩니다. 그 결과, 각 노드가 동일하며, 코드 및 구성의 변경 사항은 Cloud Manager 파이프라인을 실행해야만 전역적으로 수행할 수 있습니다.
